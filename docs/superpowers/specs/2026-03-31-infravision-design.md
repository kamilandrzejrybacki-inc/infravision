# InfraVision — Design Specification

> **Purpose**: A statically generated, interactive infrastructure map for the homelab. Shows physical topology, services, and dependencies across hosts with drill-down detail panels and filtering. Deployed to GitHub Pages.

---

## 1. OVERVIEW

InfraVision renders the homelab as a hybrid layered-host + connection-graph diagram. Physical hosts appear as containers holding their services, network zones group hosts by subnet, and dependency arrows show how services connect across hosts. Three data sources feed the map: NetBox (physical layer), Ansible (Docker services), and ArgoCD (K8s workloads).

### Key Properties

- **Static site** — React app built with Vite, served from GitHub Pages. No runtime backend.
- **Data-driven** — A single `infravision-data.json` file is generated by Prefect, committed by n8n, and consumed at load time.
- **Interactive** — Search, layer toggling, tag filtering, dependency highlighting, and click-to-inspect detail panels.

## 3. DATA PIPELINE

### 3.1 n8n Workflow: `infravision-update`

**Trigger**: Cron schedule (e.g., every 6 hours) or manual webhook.

**Steps:**
1. POST to Prefect ETL API `/api/flows/infravision-generate/runs` to trigger the flow
2. Poll for flow run completion (or use webhook callback)
3. Receive `infravision-data.json` as flow output
4. Commit the JSON file to the `infravision` GitHub repo (main branch)
5. GitHub Pages detects the push and rebuilds

### 3.2 Prefect Flow: `infravision-generate`

A Prefect flow with five tasks, four of which run in parallel:

```
                         ┌─────────────┐
                         │   trigger   │
                         └──────┬──────┘
           ┌────────────────┬───┼───┬────────────────┐
           ▼                ▼       ▼                ▼
   ┌───────────────┐ ┌────────────┐ ┌──────────────┐ ┌────────────────┐
   │ query_netbox  │ │parse_ansible│ │query_argocd  │ │query_prometheus│
   └───────┬───────┘ └─────┬──────┘ └──────┬───────┘ └───────┬────────┘
           └────────────────┴───────┴──────────────────┘
                                    ▼
                            ┌───────────────┐
                            │   normalize   │
                            └───────┬───────┘
                                    ▼
                            ┌───────────────┐
                            │  emit JSON    │
                            └───────────────┘
```

#### Task: `query_netbox`

Queries NetBox REST API for:
- **Devices** (`/api/dcim/devices/`): hostname, role, platform, status
- **Interfaces** (`/api/dcim/interfaces/?device={id}`): interface names, types
- **IP Addresses** (`/api/ipam/ip-addresses/?device={id}`): assigned IPs
- **Prefixes** (`/api/ipam/prefixes/`): network segments (CIDRs)

Returns: list of raw host records with IPs and network zone assignments.

**Auth**: NetBox API token from Vault (`secret/homelab/netbox`).

#### Task: `parse_ansible`

Clones or reads the local ansible repo and parses:
- **Inventory files** (`*/inventory/hosts.yml`): host-to-playbook mapping
- **Playbook vars** (`*/vars/main.yml` or `vars_prompt` defaults): ports, images, service names
- **Docker Compose templates** (`*/templates/docker-compose*.j2`): container definitions, port mappings, image refs, volume mounts
- **Role metadata** (`*/meta/main.yml`): dependencies between playbooks

Returns: list of Docker service records with host, ports, image, description, and playbook path.

**Reconciliation with live state**: Ansible playbooks use `vars_prompt` for interactive setup, so YAML defaults may not reflect actual deployed values. Rather than maintaining manual overrides, the `parse_ansible` output is reconciled against live container metadata from Prometheus (see `query_prometheus` task below). Prometheus container labels (collected by Alloy agents on each host) provide actual running image tags, exposed ports, and container names. Where Prometheus data conflicts with Ansible defaults, Prometheus wins — it reflects ground truth.

**Mapping logic**:
- Each playbook directory under `automation/`, `monitoring/`, `files/`, `security/`, `infrastructure/`, `ai/`, `dev-tools/`, `desktop/` maps to one or more services.
- The playbook's inventory `hosts:` field determines which host the service runs on.
- Tags are derived from the playbook's parent directory (e.g., `monitoring/grafana-stack-setup` → tag `monitoring`).

#### Task: `query_argocd`

Queries ArgoCD REST API for:
- **Applications** (`/api/v1/applications`): app name, namespace, sync status, health status, source (Helm chart repo + chart name)
- **Resources** per app: deployed resources (Deployments, Services, etc.)

Returns: list of K8s service records with namespace, Helm chart, sync status, and health.

**Auth**: ArgoCD API token from Vault (`secret/homelab/argocd`).

#### Task: `query_prometheus`

Queries Prometheus for live container metadata collected by Alloy agents on each host:

- **Container images**: `container_info` or `container_last_seen` metrics with `image` label — actual running image:tag
- **Container names**: `name` label on container metrics — maps to service identity
- **Exposed ports**: `container_network_tcp_usage_total` or similar metrics with port labels
- **Host mapping**: `instance` or `node` label — which host the container runs on

Query examples:
```promql
# Get all running containers with their images per host
group by (instance, name, image) (container_last_seen)

# Get container port bindings
group by (instance, name, port) (container_network_tcp_usage_total)
```

Returns: list of live container records with actual image, ports, and host. Used by `normalize` to reconcile against Ansible defaults.

**Auth**: Prometheus/Mimir endpoint, no auth needed (internal network).

#### Task: `normalize`

Merges the three raw datasets into the unified schema (Section 4). Responsibilities:
- Deduplicate services that appear in both Ansible and ArgoCD (e.g., n8n-workers)
- Resolve host references (NetBox device → Ansible inventory host → ArgoCD target cluster)
- Build the `connections` array from Ansible role dependencies + explicit dependency mappings
- Generate `quickLinks` by constructing URLs from known patterns:
  - Web UI: `https://{service}.{domain}` (from Caddy routes)
  - Grafana: dashboard UID lookup from `grafana-dashboards` repo
  - NetBox: device URL from NetBox API response
  - Ansible: GitHub link to playbook directory
  - ArgoCD: ArgoCD UI link for K8s apps
- Assign tags from Ansible playbook categories

#### Task: `emit`

Serializes the normalized data to JSON and returns it as the flow output.

## 4. DATA SCHEMA

The Prefect flow outputs a single JSON file consumed by the React frontend at load time.

```typescript
interface InfraVisionData {
  metadata: {
    generated_at: string;       // ISO 8601 timestamp
    sources: {
      netbox: string;           // NetBox URL
      ansible: string;          // Ansible repo commit SHA
      argocd: string;           // ArgoCD URL
    };
  };
  zones: NetworkZone[];
  hosts: Host[];
  services: Service[];
  connections: Connection[];
  tags: string[];               // All unique tags for filter UI
}

interface NetworkZone {
  id: string;                   // "primary", "nas"
  cidr: string;                 // "192.168.0.0/24"
  label: string;                // "PRIMARY NETWORK"
  hostIds: string[];            // Host IDs in this zone
}

interface Host {
  id: string;                   // "lw-c1"
  label: string;                // "lw-c1"
  ip: string;                   // ".107" (short form for display)
  fullIp: string;               // "192.168.0.107"
  zone: string;                 // Zone ID
  color: string;                // HSL string from palette
  tags: string[];
  netboxUrl?: string;           // Link to NetBox device page
  grafanaDashboard?: string;    // Link to host Grafana dashboard
}

interface Service {
  id: string;                   // "svc-n8n"
  label: string;                // "n8n"
  description: string;          // "Workflow automation platform"
  hostId: string;               // "lw-n1"
  type: "docker" | "k8s" | "native";
  ports: number[];
  image?: string;               // Docker image ref
  chart?: string;               // Helm chart name
  namespace?: string;           // K8s namespace
  syncStatus?: "synced" | "out-of-sync" | "failed";
  dependencies: string[];       // Service IDs
  tags: string[];
  quickLinks: QuickLink[];
  ansiblePlaybook?: string;     // GitHub link to playbook dir
  argocdApp?: string;           // ArgoCD UI link
}

interface QuickLink {
  label: string;                // "Open Web UI"
  url: string;
  icon: string;                 // Emoji
}

interface Connection {
  source: string;               // Service or host ID
  target: string;               // Service or host ID
  label?: string;               // "Bull queue", "USB-eth"
  type: "dependency" | "physical";
}
```

### Host Color Palette

| Host | Color | HSL |
|------|-------|-----|
| lw-c1 | Red | `hsl(0, 65%, 55%)` |
| lw-n1 | Amber | `hsl(35, 80%, 55%)` |

## 5. FRONTEND

### 5.1 Foundation

The frontend is based on the existing `homelab-navigator` repo (built with Loveable). It uses:

- **React 18** + **Vite** (build tooling)
- **@xyflow/react 12** (React Flow — canvas, nodes, edges, minimap)
- **Tailwind CSS** + **shadcn/ui** (styling and UI components)
- **TypeScript** throughout

The repo is forked/migrated into this `infravision` repo as the starting point.

### 5.2 Existing Implementation (from homelab-navigator)

Already built and functional:
- React Flow canvas with custom node types: `zone`, `host`, `service`, `k8sCluster`
- Sidebar (240px) with search, layer toggles, tag filter chips, host checkboxes
- Detail panel (320px) with service metadata, dependencies, quick links
- Layout algorithm computing explicit positions (no auto-layout)
- Dependency highlighting via React Context (BFS graph traversal)
- Dark theme color system matching the design spec
- Mock data with 4 hosts, 18 services, 2 zones, 8 dependencies

### 5.3 Changes Needed

#### 5.3.1 Edge Rendering

The existing implementation does not render dependency edges. Add:

- **Smoothstep edges** for service dependencies (animated dashed, amber `hsla(35, 70%, 55%, 0.6)`)
- **Solid edges** for host-to-host physical connections (`hsla(220, 20%, 45%, 0.4)`)
- **Edge labels** for physical links (e.g., "USB-eth") with dark background pill
- Edges render below nodes (`zIndex: 0`)
- Source/target handles on host container edges, not individual service nodes

Edge behavior on hover (already partially implemented via highlight context):
- Hovered service's dependency chain: opacity 1, strokeWidth 2
- All other edges: opacity 0.15
- Non-related nodes: opacity 0.4
- Transition: 0.2s ease

#### 5.3.2 Data Loading

Replace static mock import with JSON fetch:

```typescript
// src/data/infrastructure.ts
export async function loadInfrastructureData(): Promise<InfraVisionData> {
  const response = await fetch('/infravision-data.json');
  return response.json();
}
```

- Load JSON on app mount via TanStack Query (already configured but unused)
- Show a loading skeleton while fetching
- Show the `metadata.generated_at` timestamp in the sidebar footer
- Fall back to bundled mock data if fetch fails (development mode)

#### 5.3.3 Add lw-main Host

The mock data is missing `lw-main`. The Prefect flow will produce it from NetBox, but the frontend must handle 5 hosts (the layout algorithm already supports dynamic host counts).

#### 5.3.4 GitHub Pages Deployment

Add a GitHub Actions workflow to the infravision repo:

```yaml
# .github/workflows/deploy.yml
on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci
      - run: npm run build
      - uses: actions/deploy-pages@v4
```

The `infravision-data.json` file lives in `public/` so it's included in the Vite build output.

## 6. FRONTEND DESIGN RULES

The full visual specification is defined in the companion document `InfraVision_Design_Specification.md`. Key rules summarized here for reference:

### Layout Constants

| Constant | Value | Purpose |
|----------|-------|---------|
| `SIDEBAR_WIDTH` | 240px | Fixed left sidebar |
| `DETAIL_PANEL_WIDTH` | 320px | Slide-in right panel |
| `HOST_NODE_WIDTH` | 260px | Fixed width per host container |
| `HOST_GAP_X` | 60px | Between host containers |
| `NETWORK_ZONE_PADDING` | 60px | Inner padding of zone background |
| `SERVICE_NODE_HEIGHT` | 32px | Each service row |
| `SERVICE_NODE_PADDING` | 12px | Between service rows |

### Anti-Patterns (from design spec)

1. Never use `fitView` without explicit positions
2. Never place services as free-floating nodes — always nest inside host groups
3. Never use bezier edges — use smoothstep
4. Never make host containers auto-width — fixed 260px
5. Never use auto-layout libraries (dagre/elk) — use the explicit positioning algorithm
6. Never put text smaller than 11px
7. Never render more than 8 services per host without internal scroll

### Typography

- Host names: JetBrains Mono, 14px, 600 weight, host accent color
- Service names: Inter, 13px, 400 weight
- Zone labels: Inter, 11px, 500 weight
- Detail heading: Inter, 22px, 700 weight

## 7. DEPLOYMENT & OPERATIONS

### Build Pipeline

```
n8n trigger (cron / manual)
  → Prefect "infravision-generate" flow
    → query_netbox (parallel) ─┐
    → parse_ansible (parallel) ─┼─→ normalize → emit JSON
    → query_argocd (parallel) ─┘
  → n8n receives JSON
  → n8n commits to infravision repo (public/infravision-data.json)
  → GitHub Actions builds Vite app
  → GitHub Pages serves static site
```

### Access

The site is served from GitHub Pages at the repo's configured Pages URL. Optionally, a Caddy route can proxy to it for a custom domain (e.g., `infravision.yourdomain`).

### Update Frequency

- Default: every 6 hours via n8n cron
- Manual: trigger via n8n webhook or UI
- On infrastructure change: optionally trigger from Ansible playbook post-tasks or ArgoCD webhook

### Secrets Required

| Secret | Vault Path | Purpose |
|--------|-----------|---------|
| NetBox API token | `secret/homelab/netbox` | Query devices, IPs, prefixes |
| ArgoCD API token | `secret/homelab/argocd` | Query K8s applications |
| GitHub token | `secret/homelab/github` | Commit data file to repo |

## 8. SCOPE BOUNDARIES

### In Scope

- Static infrastructure map with interactive exploration
- Three-source data pipeline (NetBox + Ansible + ArgoCD)
- Layer toggling (physical / services / K8s)
- Search and tag filtering
- Dependency edge rendering
- Detail panel with quick links to operational tools
- GitHub Pages hosting
- Automated data refresh via n8n + Prefect

### Out of Scope

- Real-time health monitoring (Grafana handles this)
- Service management actions (start/stop/restart)
- Configuration editing
- Historical data or change tracking
- Authentication (GitHub Pages is public or org-scoped)
- Mobile-optimized layout
